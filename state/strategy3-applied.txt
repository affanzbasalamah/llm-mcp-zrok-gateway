## Routing reality check

  With open_ai pointed at litellm, anthropic pointed at api.anthropic.com, and no local slot:
    model="gpt-oss-120b"             → gpt-* prefix → open_ai → litellm → OpenRouter → gpt-oss-120b   ✓
    model="claude-haiku-4-5"         → claude- prefix → anthropic → api.anthropic.com                 ✓
    model="claude-3-5-haiku-latest"  → claude- prefix → anthropic → upstream 404 (model retired)      ✗
    model="hermes-405b"              → no prefix match → local slot (unconfigured) → 400              ✗
    model="anything-else"            → local slot (unconfigured) → 400                                ✗

  Implication: partner tier as configured (hermes-405b) will hit a routing
  error. To make it work end-to-end, either:
    a) Add a litellm alias mapping a gpt-*-prefixed name to hermes-405b
       (e.g. model_name: gpt-hermes-405b → openrouter/.../hermes-405b)
       and update partner.allowed_models accordingly, or
    b) Configure the local slot in llm-gateway with a sidecar that injects
       the bearer header into litellm-bound requests (Strategy 1 pitfall).

## Client access — tailnet-bound systemd unit (added 2026-05-28 UTC)

  The llm-gateway share is private, so it has no public URL: something must
  *dial* it. The zrok controller API is loopback-only (127.0.0.1:18080) and is
  not proxied by the :443 frontend, so off-box machines can't run their own
  zrok client against this controller without further exposure work.

  Chosen path: let the VM be the zrok client. A systemd unit runs a persistent
  `zrok2 access private` dial and re-binds it onto the VM's Tailscale interface,
  so any tailnet peer reaches the gateway as a plain HTTP endpoint — no zrok
  install, no enable token, no public surface; data plane still rides Ziti.

  Unit (tracked in repo at systemd/zrok-access-llm-gateway.service):
    /etc/systemd/system/zrok-access-llm-gateway.service   (root:root 644)
    ExecStart: zrok2 access private llm-gateway --bind 100.74.151.2:8800 --headless
    User=mcp-gw (the Dial identity; llm-gw owns the Bind side of the share)
    After=tailscaled.service + StartLimitIntervalSec=0 + Restart=on-failure
      → keeps retrying until the tailnet IP exists (boot ordering / TS flaps)
    Hardening mirrors mcp-gateway.service.

  Client usage (from any tailnet peer, e.g. a Mac — no zrok client needed):
    curl -sS -X POST http://100.74.151.2:8800/v1/chat/completions \
      -H "Authorization: Bearer <TIER_KEY>" \
      -H "Content-Type: application/json" \
      -d '{"model":"gpt-oss-120b","messages":[{"role":"user","content":"hi"}]}'
    Or point any OpenAI SDK at base_url http://100.74.151.2:8800/v1 + a tier key.

  Manage:  sudo systemctl {status,restart,stop,disable} zrok-access-llm-gateway

  Note: 100.74.151.2 is this VM's Tailscale address — environment-specific.
  Tailnet ACLs must permit peers to reach :8800 (default policy allows it).

  Edit /etc/llm-gateway/config.yaml per the patterns in
  docs/llm-gateway-routing-strategies.md, edit /etc/llm-gateway/env for any
  new keys, restart the service. No persistent state to migrate; the zrok
