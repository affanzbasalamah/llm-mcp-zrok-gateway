## Phase 2.1 snapshot caveat (DISCOVERED in Phase 5.8)

  The pre-flight snapshots captured in Phase 2.1 used the ziti CLI's
  default page limit (10 items) and were truncated. Phase 5.8 diff
  exposed this — actual production state had:

    identities:        13 vs snapshot's 10 (missed "Default Admin", and 2 others)
    service-policies:  17 vs snapshot's 10 (7 truncated)
    configs:           17 vs snapshot's 10 (7 truncated)
    services / ERPs / SERPs / config-types / edge-routers / auth-policies — all correct

  Phase 5.8 refreshed the snapshots with `limit 1000`; originals are
  preserved as `*.before-truncated.json`. Phase 2.1 logic in the
  teardown-ziti.sh script always uses `limit 1000` (was correct from
  the start) so rollback is unaffected.

## Rollback (Phase 5 reversal)

  sudo systemctl disable --now zrok2-frontend zrok2-controller
  sudo apt purge -y zrok2 zrok2-controller zrok2-frontend
  sudo apt autoremove -y
  sudo rm -rf /var/lib/zrok2-{controller,frontend} /etc/zrok2 \
              /etc/systemd/system/zrok2-{controller,frontend}.service.d
  sudo userdel zrok2-controller; sudo userdel zrok2-frontend
  sudo groupdel zrok2-tls 2>/dev/null

  # External Ziti cleanup (REQUIRED — see Phase 2.6):
  ~/zrok-rollout/teardown-ziti.sh

  # zrok2 database:
  sudo -u postgres dropdb zrok2; sudo -u postgres dropuser zrok2
